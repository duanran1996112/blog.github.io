React 组件函数式声明

函数式声明的特点：

  1. 组件不会被实例化，使得渲染性能得到提升。
  2. 由于无法实例化，组件内部 this 指向不对，无法使用 this.refs 等方法。
  3. 组件内没有生命周期，无法访问生命周期中的各个方法。
  4. 组件内没有自己的 state，通过 props 来控制组件的展现，相同的 props 得到相同的结果，复用性强，不会有副作用，不会污染全局。

函数式声明的应用场景：一些 UI 组件（按钮，输入框等）采用函数式封装，复用性强，无污染。

React 高阶组件（HOC）

利用函数式编程的思想，类似于函数中高阶函数，接受一个组件作为参数，输出一个新的组件，在这个过程中，可以修饰组件的 props 、 state、方法等，使代码看起来更加优美，复用性强。应用场景： UI 组件的一些二次封装，基础组件的二次封装、react-redux 中 connect 等。

高阶组件的声明方式：

  1. 属性代理

    属性代理可以实现：

    • 操作 props，返回新的 props。
    • 通过 refs 访问组件实例。
    • 提取 state，将组建转化为受控组件。
    • 对传入组件进行包裹。

  2. 反向继承

    反向继承可以实现：

    • 渲染劫持，过滤掉一些不想渲染的节点。
    • 条件渲染，通过 props 属性判断是否做出渲染。

注意：反向继承相对于代理属性更加凶狠一点，会改变组件一些内部的元素，反向继承典型应用 MOBX，代理属性典型应用 REDUX。

Mixin 与 HOC

高阶组件（HOC）属于函数式编程思想，对于被包裹的组件时不会感知到高阶组件的存在，而高阶组件返回的组件会在原来的组件之上具有功能增强的效果。而 Mixin 这种混入的模式，会给组件不断增加新的方法和属性，组件本身不仅可以感知，甚至需要做相关的处理（例如命名冲突、状态维护），一旦混入的模块变多时，整个组件就变的难以维护，也就是为什么如此多的 React 库都采用高阶组件的方式进行开发。
